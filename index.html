Lily

A programming language focusing on expressiveness and safety.

import sys
sys.argv.map(String.parse_i)
        .reject{|r| r.is_none() }
        .map{|m| m.unwrap() }
        .fold(0, {|a, b| a + b})
        |> print

Safety through types

Every variable has to have a starting value. Every expression has a known type. Because the type system is not optional, Lily knows what every function expects and should produce. Lily's type system exists to aid the programmer, not get in the way.

To further this notion, Lily provides Option for values that may be None.

Have a function which may fail, but want to give a message? Use Either. Exceptions should be exceptional, not everyday.

Embeddable

The interpreter does not write any global data. This allows multiple interpreters to exist at the same time without worrying about them colliding into each other.

Furthermore, Lily is frequently checked with valgrind to make sure that startup and shutdown do not leak memory. This means that you can embed Lily without fear of slowly leaking memory over time.

Lily can also be extended to hook into data of yours (this is how the postgres and apache modules currently work).

Fast turnaround

Most statically-typed languages are also compiled one. Static typing allows those compilers to generate very fast code, but at the cost of time.

Lily is an interpreted language made in hand-crafted, artisinal C. It's fast. It can churn through over 1,000 lines in less than a second.

A fast turnaround time not only encourages experimentation, but it means that you won't end up designing your modules around reducing build times.

Low memory use

Most interpreted languages have to dynamically allocate a lot of memory just to get started. A few hundred KB, or even 2 MB isn't unheard of.

Allocating memory takes time. One way that Lily cuts down on memory (and startup time) is to not dynamically allocate memory for functions and (some) functions that you aren't going to use.

Lily can do this, because it's statically typed, and knows for sure what will and won't be used. This technique, termed dynaloading, allows Lily to print 'Hello world' in less than 10 KB of memory.

Type inference

Strong static typing goes hand-in-hand with good inference. Lily knows that [1, 2, 3] is a List[Integer].

Lily requires that functions explicitly specify their input and output types. But, aside from that, most typing is optional.

You can thus change your code without having to rewrite type annotations. Types belong behind the scenes, not in your face.

Built-in templating

Lily can be run from a server and used as a templating language. When used as a templating language, code is between <?lily ... ?>, and everything else is code.

Code that is imported is expected to always be in code mode. No more importing something that accidentally sends headers. import is also namespaced, among many other things.
